# Decision 0001 — Program Governance

วันที่: 19 กรกฎาคม 2026
สถานะ: Accepted

## บริบท

งานปรับระบบครอบคลุมหลายโมดูลและมีความเสี่ยงต่างกัน ตั้งแต่ UI, Master data,
Import, ลายเซ็น, เอกสาร ไปจนถึง Permission หากรวมไว้ใน PR เดียวจะตรวจสอบ
ย้อนกลับ และระบุสาเหตุของปัญหาได้ยาก

## การตัดสินใจ

- แยกงานเป็นเฟสและ Pull Request ตามขอบเขต
- แจ้งก่อนแก้ไฟล์ทุกครั้ง
- ปิดเฟสหลังผู้ใช้ตรวจและยืนยันเท่านั้น
- ทุกเฟสต้องอัปเดตเอกสารสถานะ หลักฐานการทดสอบ Migration และ Rollback
- เก็บ Permission action inventory ระหว่างพัฒนา แต่รื้อ Permission หลังฟังก์ชันทั้งหมดเสร็จ
- ระหว่างรอ Permission redesign ใช้สิทธิ์เดิมและ minimum authorization ที่ปลอดภัย
- งาน Document ต้องผ่าน Specification และ Mockup approval ก่อนสร้าง shared engine

## ผลตามมา

ข้อดี:

- ลดการเปลี่ยน Permission ซ้ำขณะ Requirement ยังเคลื่อน
- ตรวจและ Rollback ได้เป็นส่วน
- เอกสารการตัดสินใจไม่สูญหายไปกับบทสนทนา
- ใช้ Claude หรือ reviewer อิสระตรวจจุดเสี่ยงเป็นช่วงได้

ข้อควรระวัง:

- สิทธิ์ชั่วคราวอาจกว้างกว่าสภาพสุดท้าย จึงห้ามขยายการเข้าถึงจากของเดิม
- API ใหม่ยังต้องมี owner/admin check ขั้นต่ำ ห้ามรอ Permission phase โดยไม่มี authorization
- ต้องบันทึก Action ของแต่ละฟังก์ชันทันทีเพื่อไม่ให้ตกหล่นตอน Migration
