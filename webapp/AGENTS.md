# กฎของโปรเจกต์นี้

## ปุ่ม "แก้ไข" ต้องเปิดฟอร์มตัวเดียวกับตอนสร้าง

ห้ามเขียนฟอร์มแก้แยกอีกชุด **ยกเป็น component เดียวให้สองทางเรียกใช้**

**ทำไม** — พอเป็นคนละชุด มันเพี้ยนหากันเสมอ และแต่ละฝั่งจะขาดคนละอย่างโดยไม่มีใครรู้ ของจริงที่เคยเกิด:

- **งาน** (ก่อน #478): โมดัลสร้างไม่มีช่อง "สถานะ" · ฟอร์มแก้ไม่มี ด่วน/สำคัญ, ผูกดีล, มอบหมาย → หน้ารายละเอียด **แก้ 3 อย่างนั้นไม่ได้เลย** ไม่ใช่เจตนา แค่ลืม
- **ลูกค้า** (ก่อน #481): ป้ายเบอร์โทรคนละคำ, ลำดับช่องคนละแบบ, ความกว้าง `col-span-2` vs `col-span-2 sm:col-span-1` → **จอมือถือสองฟอร์มแสดงคนละหน้าตา**
- **สินค้า** (ก่อน #482): `EditProductModal` เขียนคอมเมนต์ไว้เองว่า *"mirrors the add product form so both forms feel like one system"* = รู้ตัวว่าก๊อป แต่ไม่มีกฎให้ยึด; เพิ่มช่องสูตร (mig 0112) ต้องแก้ 2 ไฟล์ให้ตรงกันด้วยมือ

**ก่อนเพิ่มช่องใหม่ให้ entity ไหน** เช็คก่อนว่าสร้าง/แก้ ใช้ component เดียวกันหรือยัง — ถ้ายัง **ยกออกมารวมก่อน แล้วค่อยเพิ่มช่อง**

**ตัวอย่างที่ทำแล้ว** (ลอกแพตเทิร์นได้เลย):
`DealFormFields` · `PoForm` · `pm/TaskFormModal` · `database/CustomerForm` · `database/ProductForm` · `users/page.js` (`UserFields`)

**ต่างกันได้แค่ "โหมด" ผ่าน props — ไม่ใช่คนละไฟล์** เช่น
- สร้าง = ไม่มีช่องสถานะ (ของใหม่เริ่มที่ค่าตั้งต้นเสมอ) / แก้ = มีสถานะ + ช่องเหตุผลที่เกี่ยวข้อง
- `showTeams` / `canEditTeams` — ช่องที่ API เปิดให้เฉพาะบางสิทธิ์
- `factoryPrice="input" | "readonly"` — ค่าที่ต้องแก้ผ่าน action แยกเพราะกระทบข้อมูลอื่น
- `lockOf(row)` — แถวที่ผูกข้อมูลปลายน้ำแล้ว ล็อกไม่ให้แก้

**ข้อยกเว้นต้องมีเหตุผลด้านข้อมูล ไม่ใช่ความสะดวก และต้องคอมเมนต์กำกับ** เช่น หน้าแก้ PO ไม่มีปุ่มสลับ ชิ้น/ลัง เพราะ DB เก็บเป็นชิ้น — เลขที่โหลดมาจะถูกตีความใหม่เป็นลังแล้วข้อมูลเพี้ยนเงียบ ๆ
